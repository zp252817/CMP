LTPO 动态刷新率方案

LTPO 动态刷新率方案

参考flutter接入LTPO，当前仅覆盖抛滑场景（fling），动画场景暂未接入。本方案梳理调用链路、关键阈值与可观测点，并给出落地建议。

1. 目标

  • 抛滑时依据滑动速度动态投票期望刷新率，兼顾流畅与功耗。
  • 链路清晰、易观测、可按机型/实验快速调优。

2. 调用链路（现状）

1) 入口：滚动层（Compose Fling）

  • Scrollable.updateLTPORefreshRate(velocity)
  • 路径：compose/foundation/.../Scrollable.kt 972-986
  • 将 px/s → mm/s（基于 PPI、缩放因子），然后 setScrollVelocityForLTPO(velocityMmPerSec.toDouble())。
  • 低速阈值：abs(velocityPxPerSec) < 1f 直接 return（当前未投票）。

2) Kotlin → Native 桥接

  • setScrollVelocityForLTPO 透传到 C++（cinterop/JNI），将 mm/s 传给底层。

3) Native 侧决策

  • UpdateExpectedRefreshRateByVelocity(double v_mm_per_s)
  • 路径：OHRender/OHRender/src/oh/RenderNodeImplC.cpp 56-63
  • g_expected_refresh_rate = ChooseRefreshRateByVelocity(v_mm_per_s)
  • 组装 OH_NativeVSync_ExpectedRateRange，调用 OH_NativeVSync_SetExpectedFrameRateRange(g_vsync_handle, &range) 投票。

4) 平台/硬件侧

  • VSYNC 接口提交期望帧率区间，最终帧率由系统/硬件 LTPO 逻辑决策。

3. 原理摘要（抛滑）

  • 速度大 → 期望高刷（提升流畅）。
  • 速度小/停 → 期望降刷（节能）。
  • 期望值通过“投票”提交，系统可能按多方投票/功耗策略最终取值。

4. 映射档位（参考 ArkUI 投票机制）

参考点：ArkUI/NativeVSync 提供的“期望帧率投票”接口 OH_NativeVSync_SetExpectedFrameRateRange。

当前实现的“速度→期望刷新率”映射来自 RenderNodeImplC.cpp 的 ChooseRefreshRateByVelocity(v_mm_per_s)（单位：mm/s）。

4.1 速度 → 期望刷新率（mm/s → Hz）

抛滑速度(mm/s)期望刷新率（Hz）备注
(0 < v < 10)60低速/接近停下
(10 < v < 46)72中低速
(46 < v < 77)90中速
(77 < v < 800)120高速抛滑，优先保证流畅
(v > 800)90极端高速场景避免一直锁 120Hz（当前策略）

4.2 投票接口

  • OH_NativeVSync_ExpectedRateRange range = { min, max, expected }
  • 当前实现固定：min=60、max=120、expected=上表映射结果
  • 调用：OH_NativeVSync_SetExpectedFrameRateRange(g_vsync_handle, &range)

5. 多元投票源头（参考 ArkUI，说明“谁在投票”）

5.1 典型投票源头列表

1) 触屏/输入侧投票（Input / Touch）

  • 触发时机：用户按下/滑动开始、持续交互期间。
  • 投票目的：提升交互首帧、滑动跟手的体验，立即投 120Hz，保障交互首帧和滑动的性能体验。
  • 投票接口：由系统内部完成（同样会影响最终帧率仲裁）。

2) 原生组件 / 原生动画投票（ArkUI Native Components/Animations）

  • 触发时机：ArkUI 原生组件在自身滚动/动画期间（例如原生列表滚动、系统动画）。
  • 投票目的：保证原生侧动画/滚动的流畅度。
  • 投票接口：组件/框架可通过 ArkUI 的 displaySync/NativeVSync 能力投票（实现细节因模块不同而异）。

4) 视频播放投票（Video/Media）

  • 触发时机：视频播放和暂停期间（尤其高帧率/高运动量内容）。
  • 投票目的：
  • 视频播放时匹配视频帧率与显示刷新率（例如 60fps/120fps），减少抖动/抽帧。
  • 视频暂停时，可以退回到较低的屏幕刷新率（ 60Hz），以节省能耗。
  • 投票接口：未知（会影响最终帧率仲裁）。

5.2 多票源如何“合并”

  • 合并者：ArkUI/系统层（非CMP决定）。
  • 合并原则（常见理解）：
  • 多方同时投票时，系统可能选择满足体验的更高档位（具体策略由系统决定，不保证“取最大/取最近”等固定规则）。
  • 当高优先级投票源（输入/视频/原生动画）存在时，Compose 的投票可能被覆盖。

6. 现状风险/观察点

  • 低速 return：compose运行时中，abs(velocity) < 1f 直接返回， 返回后是否设置为60fps？
  • PPI 依赖：需要获取手机PPI，通过ppi将vp -> mm 。
  • 防抖缺失：每帧都参与投票，高频投票可能增加开销。
  • 取消投票时机 抛滑结束后，是否可以投0fps，取消投票

7. 动画场景（未接入，后续规划）

  • 可复用同一速度→帧率映射，但输入为“动画驱动力度/速度”而非滑动速度。
  • 动画结束后应取消投票，避免高刷滞留。
  • 动画也复用同一档位配置路径，减少分叉策略。

8. 测试trace和日志

  • Kotlin：在 updateLTPORefreshRate 增加trace和日志 & PPI 默认值；低速阈值改为可配置。
  • C++：在 UpdateExpectedRefreshRateByVelocity 增加日志；ChooseRefreshRateByVelocity 档位表外置或集中定义；PPI 异常、映射异常回退。
  • 可选：加入简单防抖（时间窗口或速度变化阈值）。

9.功能验收与测试

  • 场景：慢速/中速/高速抛滑，观察期望刷新率是否随速度阶梯变化。
  • 指标：
  • 流畅度：高速滑动时帧率可升档。
  • 功耗：停止/低速时能及时降档。
  • 日志：能看到速度、档位、投票值，若可查询生效帧率则对齐验证。
  • 机型/档位覆盖：验证典型 LTPO 档位（如 1~10Hz 60 90 / 120）。

10. 多元投票专项测试场景

为验证多源投票的互相影响与仲裁，建议在同一机型上分场景、分阶段执行，并记录期望/生效帧率日志：

1) 基线场景：单源抛滑

  • 仅触发 Compose 抛滑（其他源静默），验证速度阶梯 → 档位映射是否正确。

2) 输入优先生效场景

  • 触屏滑动开始（输入侧高刷投票），随后保持轻微滑动，观察 Compose 低速票与输入高刷票的仲裁结果。
  • 预期：若输入票优先级高，可能维持高刷；输入结束后应允许降档。

3) 原生组件动画并行

  • 启动 ArkUI 原生列表滚动或系统动画，同时触发 Compose 抛滑，观察两路投票合并后帧率走向。
  • 预期：高优先级原生动画存在时，Compose 票可能被覆盖或只在动画结束后生效。

4) 视频播放并行

  • 播放高帧率视频（如 60/120fps），同时做轻微 Compose 滑动，观察视频票与滑动票的合并。
  • 预期：视频侧可能维持较高帧率；滑动结束后若视频暂停，应允许降档。

5) 退票/降票验证

  • 抛滑结束、输入结束、动画结束后，检查是否有明确“降票/退票”动作（例如设置 expected=60 或 expected=0 作为退出信号）。
  • 预期：无残留高刷投票，帧率能回落。

6) 极端高速与回退

  • 制造极高速滑动（速度 ≥ 800mm/s），观察映射表中的回退到 90Hz 是否触发，确认不会长时间锁 120Hz。

7) 防抖/窗口聚合有效性

  • 如果加入防抖/聚合，验证日志中投票频率是否下降，同时档位切换的响应是否仍然及时。

记录要点：

  • 每个场景记录：速度（px/s、mm/s）、期望档位、投票接口参数（expected/min/max）、若可获取则记录实际生效帧率。
  • 观察多源并行时的仲裁结果，确认不会出现“高刷滞留”或“低刷卡顿”。
